- Introductions.
- Summary
  - Test suite issues and package identification complexities were discussed, focusing on a unified identifier approach and the necessity of standardized qualifiers for commercial software.
  - The meeting prioritized addressing issue 837, the identifier discussion, over test suite problems, due to the need to resolve issues with identifying non-registry packages like the Linux kernel and commercial software. A unified identifier approach was proposed, combining a domain-qualified method with an optional Git-based registry for registration and discovery.
  - Concern arose that the proposed identifier lacked standardized qualifiers necessary to support commercial software and replace CPE, leading to a debate on optional versus required qualifiers. A key agreement was reached to provide a standardized, optional set of common qualifiers to facilitate migration from CPE, with the possibility of per-vendor requirements defined in well-known documents.
  - Discussion clarified the utility of the PURL subpath component (using the `#` separator) for identifying specific files within large repositories, such as configuration files or modified files within jars.
- **Meeting Logistics and Code of Conduct**: The meeting commenced with a reminder that the session is being recorded, primarily to assist with creating accurate meeting minutes, and that the recording will be discarded afterwards. Participants are subject to the project's code of conduct and the rules of TC-54, which generally encourage polite and professional behavior.
- **Test Suite Issues and Blockage**: Michael Herzog raised concerns about being blocked due to unresolved issues related to the test suite, specifically concerning the existence of a validation test type and how messages relate to validation. They noted that the distinction between validation and round-trip tests is not entirely clear and emphasized that the test suite needs attention.
- **The PURL Expansion Project and Agenda Items**: Tom Alrich introduced the "PURL Expansion Project" proposal they had previously circulated, highlighting that it includes the scid work but expands beyond it. Jon Moroney inquired about file specifiers for the generic type and the origin of the hash symbol used in the GitHub type for specifying a file, noting a lack of documentation.
- **Interest in the Identifier Discussion**: Pavel Shukhman and Joshua Kugler indicated that they did not have additional agenda items, expressing primary interest in the identifier (scid) discussion.
- **Prioritizing the Identifier Discussion (837)**: Michael Herzog acknowledged that the test suite discussion would take significant time, suggesting that addressing issue 837 (the identifier discussion) should be prioritized.
- **Context and Problem of Package Identification**: Philippe Ombredanne outlined the problem that the proposed identifier type aims to solve: identifying packages that do not exist in structured registries, such as standalone open-source projects (like the Linux kernel or OpenSSL) and commercial or internal-use software. The goal is to provide a mechanism, especially after discussions with figures like Greg Kroah-Hartman from the Linux Foundation, to ensure SBOMs are actionable, which is currently difficult with the generic type or due to issues with CPE adoption.
- **Proposed Unified Approach to Identifiers**: Philippe Ombredanne proposed a unified approach for identifier types, suggesting a single type (currently called "sid" but flexible regarding the final name like "scid") that requires a namespace and allows optional segments separated by slashes. The approach combines both a domain-qualified method, similar to scid, and an optional Git-based registry for centralized or initial registration.
- **Discovery Mechanisms (Registry and .well-known/ Files)**: The unified proposal includes using a .well-known/ file served at a domain for decentralized metadata, offering an alternative to or a combination with the optional Git-based registry. Philippe Ombredanne noted that this allows for early adoption, such as by kernel.org, even if they delay publishing a .well-known/ file, and suggested that the domain registration should take precedence over the registry if both exist.
- **Concerns Regarding Commercial Software and Qualifiers**: Tom Alrich expressed concern that the scid proposal attempts to address two disparate use cases—commercial and non-package manager open-source software—in one type. Steve Springett argued that the proposal is insufficient for commercial software vendors because it lacks the necessary standardized qualifiers required to displace CPE.
- **Discussion on Optional vs. Required Qualifiers**: Pavel Shukhman, supporting a single type, argued that qualifiers should remain optional in PURL, allowing open-source projects to skip them while commercial vendors utilize them. Michael Herzog noted a recent consensus that required qualifiers are possible for a PURL type, although generally not recommended. Matt Rutkowski and Steve Springett clarified that while qualifiers are optional in the base PURL specification, certain use cases (like commercial vendors) or layered requirements (like security profiles) may necessitate their use, and the base spec should define the semantics of these common qualifiers.
- **Impact on Tooling and Proposed Observation Surface**: Michael Herzog voiced concerns that identifiers not found in a public, well-stated registry will pose a "huge hurdle" for tooling, parsing, and building validation, particularly for commercial software. Jon Moroney suggested creating an "observation surface" that would proactively look for and index well-known files to build a non-authoritative, observed registry of known names.
- **Single Discovery Approach and Technology Neutrality**: Steve Springett advocated for a single discovery approach, moving away from technology-specific solutions like Git for the identifier type and suggesting the use of a single well-known mechanism that could default to `packageurl.org/.well-known` and include the ability to redirect. Jon Moroney clarified that the mention of Git was intended as an implementation detail for file storage and history, not a dependency of the PURL type.
- **Need for Standardized Qualifiers to Replace CPE**: Steve Springett insisted that to effectively displace CPE, a standardized set of qualifiers (e.g., arch, edition, target, locale) is needed, even if some are not applicable to all open-source projects. The discussion concluded with an agreement that providing a standardized, optional set of qualifiers would offer a "happy path for migration" from CPE.
- **Proposal for Per-Vendor Requirements in .well-known/ Documents**: Philippe Ombredanne suggested that a namespace could define its own required and optional qualifiers, separate from standard ones, possibly specified within the .well-known/ file. Jon and Steve Springett found this concept of self-specifying requirements in the .well-known/ file "really interesting," as it would allow vendors (like Oracle) to precisely enumerate the qualifiers (e.g., specific editions or architectures) that apply to their products, aiding security researchers in vulnerability management.
- **Future Steps on the Identifier Proposal**: Steve Springett proposed that the existing discussion should be integrated as a modification into the original scid issue (https://github.com/package-url/purl-spec/issues/516), and the discussion about location/discovery should be moved to a separate ticket to avoid conflation. Philippe Ombredanne agreed to close the current issue (https://github.com/package-url/purl-spec/issues/837) and add the feedback to the original scid issue, but expressed difficulty in completely separating the namespace and location discussions, given that a domain-qualified namespace implies discoverability. Jon Moroney requested that Steve Springett and Matt Rutkowski provide follow-up scenarios on the issue thread regarding specific qualifiers and commercial software.
- **Avoiding Centralized Infrastructure Dependency**: Philippe Ombredanne expressed a preference for avoiding dependencies on centralized infrastructure, noting that the ease of cloning and synchronization of a Git repository means a tool developer does not need to depend on a central service being active. Steve Springett noted that CycloneDX uses Cloudflare because some libraries do not cache schemas locally, and they receive millions of requests a day for schemas, indicating the need for highly available infrastructure for some standards. Matt Rutkowski stressed that any group adopting a standard must be responsible for ensuring the connectivity and availability of what they are asking people to connect to, despite any potential discomfort.
- **Integration of Feedback and Future Plan for PR**: Philippe Ombredanne committed to integrating all the discussion feedback into their work and mentioned they would be creating a very large pull request. Jon introduced the topic of the Common Package Specification (CPS) published by the CMake group as a last-minute item that should be on everyone's radar. Philippe Ombredanne confirmed that they were already aware of and tracking CPS, which is an evolving specification from a collaboration including CCK, Mason, and Conan, discussed within the ISO C++ community.
- **Common Package Specification (CPS) and its Relevance**: The Common Package Specification (CPS) is intended as a neutral replacement for `pkg-config` that is agnostic to the build tool and primarily concerned with metadata. The goal of CPS is to facilitate the discovery of necessary components, such as libraries and headers, without dependence on a specific build environment. Philippe Ombredanne confirmed that their current work, specifically promoting a prototype that supports C/C++, will be integrated into the CPS effort, slightly modifying the original idea that their work had to be based on CPS. Philippe Ombredanne noted that the C++ community should ultimately collaborate, especially since CMake is the de facto standard for C/C++ software, used by approximately 82% of users. The underlying purpose of CPS is to resolve the problem of conveying configuration information, like dozens of flags used when building complex software such as `llama.cpp`.
- **Using Build Artifacts and PURLs for CycloneDX Generation**: Matt Rutkowski shared that they use AI to determine required flags and behaviors for target environments and different versions of libraries, suggesting a different perspective on configuration. The core goal is for CPS to provide a more generic way of managing native code configurations, and the hope is that they can provide a PackageURL (PURL) that can be injected as input to the build process to eventually generate a proper CycloneDX Software Bill of Materials (SBOM). Matt Rutkowski felt that this topic required a separate call to explore how it affects the manufacturing bill of materials, behaviors, and design-time requirements.
- **Clarification on the Use of Subpath in Package-URLs (PURL)**: Jon Moroney shifted the topic to the use of the hash or pound symbol (`#`) for specifying files in the generic Git PURL type, which they had been copying from the GitHub and Bitbucket purl types. Michael Herzog clarified that the pound symbol is the separator used for the subpath component of a PURL, which is separate from the qualifiers defined by the question mark symbol (`?`). The participants confirmed that documentation for the PURL core specification, including the subpath component, is available on the website.
- **Applications of the PURL Subpath Component**: The subpath component is useful for specifying a particular file within a repository, especially in large monorepos with hundreds of packages. Matt Rutkowski confirmed they use the subpath in the Machine Learning (ML) Bill of Materials guide to identify specific files, such as configuration files, within a model repository. Steve Springett explained that they used the subpath at their former employer to list files that had been modified within jars and to specify files that had been excluded during "tree shaking," which involves removing unused code from a component.
