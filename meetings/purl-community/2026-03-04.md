- Summary
  - The meeting participants, including Michael Herzog, Philippe Ombredanne, Jonatan Männchen, John Horan, Jon Moroney, and Tom Alrich, discussed various topics, commencing with a review of a Wall Street Journal article concerning the Gemini AI model and its potential negative influence on users.
  - Michael Herzog reviewed the agenda, and Jonatan Männchen introduced a critical issue regarding version range boundaries in the VERS specification, specifically due to the inability to losslessly convert version constraints.
  - Michael Herzog detailed the plan to develop VERS 'types' or schemas for specific package managers, such as the Elixir/Erlang ecosystem, to accommodate complex versioning rules.
  - Michael also confirmed the ongoing status of the non-package registry 'scid' proposal, updated by Tom Alrich, and the progress on the Git PURL type issue (https://github.com/package-url/purl-spec/issues/780), for which Jon Moroney requested permission to submit a pull request.
- **Meeting Agenda and Roll Call**: Michael Herzog outlined the initial agenda items, which included a website update, the scid proposal, and the Git PURL type proposals. Philippe Ombredanne conducted a roll call.
- **Version Range Discussion Topic Introduction**: The discussion turned to Jonatan Männchen's previously raised topic regarding version range boundaries (https://github.com/package-url/vers-spec/issues/57). The core issue concerned the inability to losslessly convert version constraints to and from the VERS specification, particularly around pre-release versions (alpha, beta, etc.) because ordering in semantic versioning does not allow for easily defining boundaries.
- **Ongoing Specification Issues**: Michael Herzog listed the ongoing discussions as the non-package registry/scid and the Git PURL type issues, acknowledging they were top-of-mind. Jon Moroney confirmed their focus on the Git PURL type. Tom Alrich confirmed they were satisfied with the agenda.
- **Challenges in Version Range Definition**: Jonatan Männchen explained that the version problem is visible when using SemVer due to the infinite steps possible between any two versions, making boundary definition difficult. Philippe Ombredanne concurred that an infinite number of versions can always be placed between any two versions, meaning a version range is only accurate based on the versions that exist when the range is produced.
- **Enumerated Ranges vs. Version Ranges**: Philippe Ombredanne explored the trade-off between fully enumerated ranges (lists of affected versions) and version ranges (e.g., "all versions before five"). Jon pointed out that from an operations standpoint, version ranges are more maintainable and less error-prone for human editors, than large enumerated lists for vulnerability data. Jonatan Männchen noted that current version representation in VERS is unclear, potentially leading to different results than actual package managers.
- **Elixir's Special Operators and Comparators**: Jonatan Männchen described specific issues in the Elixir world with the tilde '~' operator, which specifies ranges like "larger than this version and smaller than the next major version up". They noted that pre-releases often complicate these boundaries, and current VERS specification, as they understood it, does not allow for new operators (or comparators) necessary to map these behaviors. Jonatan Männchen suggested adding Elixir's operator but noted that this could lead to an excessive number of ecosystem-specific operators.
- **Development of VERS Types and Schemas**: Michael Herzog explained the intention for the VERS specification to include VERS "types" or schemes which can map to a package manager or ecosystem, allowing the definition of specific rules and logic. The intent is to develop a VERS type schema, similar to the existing PURL type schema. Michael Herzog suggested using the Elixir/Erlang ecosystem as a working example for prototyping the VERS type schema and addressing specific syntax and semantics.
- **Decoupling Notation from Meaning in VERS**: Michael Herzog stressed that notation provides a compact expression, but the actual meaning must be decoded at the time of use by expanding it to a set of enumerated versions, which may change over time. They affirmed that the efficiency of VERS comes from its succinct notation rather than maintaining long enumerated lists. Jonatan Männchen confirmed that the Elixir version module is often copied by other tools in the Erlang ecosystem, suggesting that an "Elixir version type" or "Hex version type" would be appropriate.
- **Status of the VERS Specification**: Michael Herzog confirmed that new comparators are not off the table, as the VERS specification is still at an early stage, targeting a v1.0.0 release by the end of the year. Philippe Ombredanne agreed, noting that the issue may not require new comparators but rather proper specification and implementation of the Elixir/Hex scheme as its own version type. Jonatan Männchen demonstrated how Elixir's special operator defines boundaries based on version segments and not just the upper bound, even when pre-releases are allowed.
- **Action Items for VERS Development**: Philippe Ombredanne requested a specific non-working example from Jonatan Männchen's property-based testing to better understand the failure. Michael Herzog confirmed they would create a template for adding a new VERS type. Jonatan Männchen committed to thinking about the Elixir type definition and expressed a goal to eventually integrate the VERS parser into the core Elixir version module.
- **Website Update and Progress**: Michael Herzog provided an update on the website (www.packageurl.org), noting improvements such as moving the tools and specifications section to "getting started" and adding drop-down menus. The next update involves reorganizing the current long narrative VERS specification into a series of documents, including drafts for the Ecma submission, and they are confident they can complete this work within the week. John Horan confirmed that Docsaurus has been productive overall.
- **scid and Commercial/Proprietary Use Cases**: Michael Herzog acknowledged the ongoing scid and Git discussions, noting that Tom Alrich had produced a simplified proposal for promoting PURL for commercial or proprietary use, which involves open source without a package registry. Philippe Ombredanne mentioned that HPE internally created custom PURL types for private software, which suggests a need for guidance on handling private packages, perhaps by introducing a "private" package type to signal to tools that the component is not publicly resolvable. Michael Herzog emphasized that an SBOM must be complete, meaning all components, including private or proprietary ones, should be accounted for.
- **Git PURL Type Update**: Jon Moroney confirmed that the Git PURL type discussion had progressed from VCS to "Git only" and from using only commits to supporting any Git reference. Jon requested permission to submit a pull request to move this forward and shared a demo project showing a PURL-in/PURL-out process intended to make vulnerability reporting as precise as possible, including commit, file, and line details.
- **Conclusion of Current Discussion and Next Steps**: Michael Herzog confirmed the conclusion of the current discussion, stating they would pick up there the following week. The participants expressed appreciation before concluding the meeting.
- **Integration of Provenance Information**: Matt Rutkowski suggested that the discussed tool would be a good plugin for Steve Springett's "studio" to integrate provenance information. Michael Herzog confirmed the potential of this integration.
- The meeting was adjourned.
