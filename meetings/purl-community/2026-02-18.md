- **Community Meeting Logistics and Agenda Setting**: Philippe Ombredanne opened the PURL community meeting, noting the recording is for transcription purposes and is subject to the code of conduct. Michael Herzog pointed out that the running notes contain a new tab called 'PURL Spec issues' to serve as a handy reference for recurring issues, distinguishing between current and strategic topics. Attendees conducted a roll call to confirm agenda items, including a discussion on non-package software, website updates, and issues with version ranges.
- **Strategy for Non-Package Software**: Steve Springett raised the strategic topic of determining the community's approach for non-package software, such as scid. They acknowledged that this is a complex issue requiring multiple sessions and suggested prioritizing more immediate concerns. This topic, along with other strategic issues, is listed in the 'PURL Spec issues' tab within the running notes for future discussion.
- **Introduction of Jonatan Männchen**: Jonatan Männchen, the Chief Information Security Officer (CISO) of the Erlang Ecosystem Foundation, introduced themself as a first-time participant. Their primary agenda item concerned a design philosophy question about encoding version ranges (VERS) encountered while implementing CycloneDX in Software Bill of Materials (SBOM) tools, particularly concerning a version range field in CycloneDX 1.7.
- **Discussion on Version Control System (VCS) PURL Type**: Jon proposed revisiting the VCS PURL type, leaning towards a specific Git type rather than a bundled type like the Maven SCM. This approach is favored for its simplicity and the potential to reuse data from Andrew Nesbitt's ecosystem project, which currently uses an ad-hoc Git type. Jon agreed to post comments on the issue to gather feedback and clarify references, acknowledging that they were unsure of the exact reference to Mr. Nesbitt's project.
- **VERS Standardization Timeline**: Philippe Ombredanne provided an update on the VERS progress, indicating that meeting the June deadline for Ecma standardization is challenging due to their commitment to grant proposals, which must be completed by early March. Philippe expressed a preference not to rush the process to avoid a "death march" and suggested aiming for a more comfortable December timeline. Michael Herzog agreed that a bad submission would be the worst outcome, supporting the goal of a December submission.
- **Realistic Timeline for VERS Specification**: Steve Springett agreed with the decision to aim for a December standardization timeline, emphasizing that a rushed, bad submission would reflect poorly on all associated standards, especially given the likelihood of an ISO fast path for VERS. They advocated for following the normal order for development and standardization, noting that having the spec available on the website will be sufficient for most users until formal standardization occurs. Steve Springett noted that this extended timeline allows time to review, solidify the specification, and address questions like those raised by Jonatan Männchen, allowing for the discovery of potential design flaws.
- **TC54 Chair Transition**: Steve Springett, currently the Chair of TC54, announced they will be leaving their organization, ServiceNow, in just under a month. This transition necessitates a new chair for TC54, as their next organization may not be an Ecma member. Steve Springett mentioned they would likely continue as an invited expert for the foreseeable future.
- **Issue with VERS Pre-releases and Ranges**: Jonatan Männchen presented an issue encountered during their VERS implementation, where fuzzing revealed that converting to VERS and back did not always yield the same results. The primary problem lies in SemVer pre-releases, which are ordered before production versions, making it difficult to use version ranges (e.g., greater than 1.1 and less than 2.0) to exclude pre-releases (e.g., 2.0-alpha) without knowing the full list of versions beforehand. Jonatan Männchen expressed concern that the current spec might not support lossless encoding or is missing elements to represent these complex constraints, especially since the VERS constraints should resolve to versions that the native package manager would resolve to.
- **Complexity of Version Ordering and Scope**: Philippe Ombredanne clarified that version ordering is entirely dependent on the scheme and its defined semantics, noting that implementations like npm, while claiming SemVer compliance, are effectively using their own specific standards. They acknowledged that the problem is difficult because the number of versions between two points is potentially infinite, which creates tension between enumerating all versions and using a compressed range. Jon mentioned that the problem of defining affected versions via ranges is a common but complex pattern in the CVE space. Jonatan Männchen departed the meeting but encouraged further discussion on the issue tracker.
- **Website Update and Governance**: Michael Herzog provided an update on the website, confirming the release of version 1.0, which is hosted on DreamHost, with GitHub.io serving as a staging environment. They announced that patch releases, primarily for new data like tool or PURL type listings, will be issued approximately once a week and try to be faithful to the spirit of SemVer. The README has been updated with basic rules against direct, generated pull requests (PRs) without a prior issue, though exceptions are made for simple fixes like typos. The staging site remains available for those interested in the latest developments, and the public site is hosted by nexB.
- **Conflict Between Spec and Schema for Required Qualifiers**: Michael Herzog raised issue 808 (https://github.com/package-url/purl-spec/issues/808), noting a conflict between the specification and the schema regarding required qualifiers, stemming from the Julia package type. The schema allows a required qualifier key, but the specification states that qualifiers are always optional. Michael Herzog believed the qualifiers should remain optional according to the specification and suggested that compliance tests should not require them.
- **Resolution on Qualifier Requirement Conflict**: Steve Springett proposed that there might not be a direct conflict, arguing that while qualifiers are optional on the spec side, specific PURL types can introduce tighter, required components, similar to how a URL requires a protocol even though a general Uniform Resource Identifier (URI) does not. Michael Herzog agreed with this interpretation but noted that the standard must be updated to clearly state that qualifiers may additionally be required depending on the PURL type, as this clarification is currently missing. All participants agreed that this clarification is the right approach and that while type-specific required qualifiers are permitted, they should be used judiciously and seen as a last resort.
- **Plan for VCS Git Type Implementation**: Jon reiterated their preference for a pure Git type for source code repositories, citing lower scope and existing internal use by Andrew Nesbitt's ecosystem project. Jon agreed to draft a PR for the next meeting, which will likely be a generic Git type, independent of hosting services like GitHub or GitLab. They expressed that their general approach is to propose the simplest possible solution and adapt it based on valid community concerns.
- **Protocol Exclusion for Git Type**: Jon and Philippe Ombredanne agreed that the proposed Git type should not include the protocol (e.g., HTTP, HTTPS) as part of the specification, because the protocol is often specific to the tool or transport and is a meaningless difference in the context of identifying the code. Philippe Ombredanne provided an example of a past mistake where various protocols were included in an SPDX-related effort, confirming the complexity that results from linking the specification to transport protocols.
- **Addressing Git-Specific Challenges**: The discussion shifted to specific Git features that the new type must address, such as the handling of branches, references (refs), and commits versus tags. Jon acknowledged that tags are very human-friendly but suggested that tools should accept tags and then convert them to the more concrete and temporally stable commit identifier for the data representation. Steve Springett added that the type will need to address Git's equivalent of SVN externals to be explicit about support or lack thereof.
- **SPDX Update and ECMA-427**: Michael Herzog mentioned that they are coordinating with Gary O’Neal regarding ECMA-427 and the current SPDX 3.0.1 draft. The draft includes a good, though not exact, copy of the Ecma pre-submission content, and they plan to ensure PURL is properly represented in the next SPDX release (3.0.2 or later).
- **Software Heritage Reference**: Philippe Ombredanne briefly introduced Software Heritage as a project that acts as a petabyte-sized Git repository. This project uses its own content addressable identifier (SWHID), which has become an ISO standard, to address content by tree or file (https://www.softwareheritage.org/software-hash-identifier-swhid/), and Jon expressed interest in researching this further.
- **Handling Git Repository Cross-Inclusion**: The group discussed the challenge of one Git repository including source code from another Git repository during build time. Jon initially leaned toward not allowing this behavior and suggested using a single Package-URL (PURL) to identify the primary repository and its problematic commit. Philippe Ombredanne noted that this issue extends beyond submodules to include Git LFS and similar systems used by platforms like Hugging Face.
- **Identifying Composed Content**: The discussion addressed whether the underlying storage for content should matter, concluding that it might not be the core issue. Philippe Ombredanne explained that these nested scenarios result in a combination of a specific commit, specific content, and a tool capable of retrieving content from elsewhere. They suggested that a full composition analysis of a Git repository might require multiple PURLs to describe the various included repositories.
- **Bounding the Problem of Inclusion**: The participants considered the practical difficulty of tracking all forms of code inclusion, noting that an attacker could simply use a `curl` command within a Git repository to download and compile external code at build time, an issue Jon described as "unbounded". Although Philippe Ombredanne suggested their charter is to address unbounded problems, Jon maintained that bounded solutions still provide value. The group agreed they need to take a formal position on supporting these nested inclusions, possibly by supporting the feature but disabling it by default unless a specific qualifier is present.
- **Future Agenda and Follow-up Tasks**: The meeting concluded after reaching the hour mark, with the next meeting (the TC54-TG2 meeting) scheduled for a week from Friday. Michael Herzog confirmed that the SCID (Software Component ID) PURL type topic is scheduled as the main discussion point for the next TG2 meeting. Jon was asked to create an issue regarding the discussion on Git repository cross-inclusion and prompt participants to provide feedback on that topic.
- The meeting was adjourned.
