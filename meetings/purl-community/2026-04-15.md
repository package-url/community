## Notes

- Meeting minutes are being kept and will be published, and the meeting will be recorded with Google Meet video and Gemini "note-taking".
- Our code of conduct (link in agenda above) applies to this meeting.
- Introductions.
- Summary
  - The team refined the PURL type proposal queue and introduced a template for new requests.
  - Participants achieved consensus to prioritize the canonical VERS enforcement pull request to eliminate normalization.
  - The group moved to adopt a validation test type to replace round trip testing. Developers agreed to implement a standardized error reporting system using specific codes and severity levels.
  - Participants identified the need for expressing abstract feature dependencies rather than package-specific requirements. The team reached a decision to prioritize developing a new schema for these capability-based dependencies.
- **Meeting Logistics and Code of Conduct**: Michael Herzog welcomed attendees to the PURL community meeting and noted that the session is being recorded using Google Meet video and that they use Gemini for note-taking. The recordings are used to generate meeting minutes and will be removed one month after the minutes are approved and archived. The ECMA code of conduct is applicable to the meeting.
- **Update on purl-spec Issues and PRs for PURL Types**: Michael Herzog provided a brief update, stating that they are currently working on purl-spec issues and pull requests, with a focus on PURL types. The team introduced three main labels for handling new and changed PURL types: `purl type: new`, `purl type: change` for existing types, and `purl type: new / refactor` for a handful of old PRs that need to be redone.
- **Status of PURL Type PR Queue**: The process for managing the queue of PURL type proposals has been cleaned up to make the criteria clearer and more efficient. Currently, there are 17 new PURL type PRs, with about a third of them considered reasonably ripe, including proposals for `git` led by John Moroney (Darakian) and a good one from `homebrew`.
- **First Request Using PURL Type Template**: The team received their first request using the new PURL type pull template for `Ansible collections`. They are beginning work on this request.
- **PR for Canonical VERS Enforcement**: Immanuel Kunz presented a PR regarding the issue of canonical VERS, where the suggestion is to strictly allow only canonical forms and eliminate normalization. Consensus seems to be that this is a good idea, and the PR includes changes to the specification, parsing instructions, and tests, including a new test that specifically rejects non-canonical VERS.
- **Discussion on Canonical VERS PR Details**: Philippe Ombredanne stated they had no immediate feedback on the canonical VERS PR but agreed that the principle is a "no-brainer". The discussion moved to refining the wording and considering if a separate section should exist outside the core standard for a flexible, recovering parser that handles non-canonical forms. Michael Herzog assigned the PR a high priority label.
- **Proposal for Reserved Private Version Schemes**: Immanuel Kunz brought up an issue (https://github.com/package-url/vers-spec/issues/17) proposing the addition of a reserved private namespace for version schemes, suggesting an `x-` prefix for organizations to experiment with private schemes in their own context. Philippe Ombredanne is not opposed in principle but would prefer alignment with how private PURLs are handled, and requested actual real-world examples to support the use case.
- **Caution Regarding x- Prefixes and Private Schemes**: Salve J. Nilsen advised that the Internet Engineering Task Force (IETF) discourages the use of `x-` prefixes due to the risk of temporary measures becoming permanent and causing difficulty. If an extension like this is introduced, a clear procedure should be established for how these examples can be incorporated or normalized into regular keywords (00:09:56). Michael Herzog clarified that the current discussion is about strictly private schemes, not for commercial vendors.
- **Next Steps for Private Version Schemes**: Immanuel Kunz committed to looking more closely into the real-world relevance and gathering examples for the private version scheme discussion. They also agreed to look into how this compares with private package types on the PURL side.
- **Discussion on Version Range Constraints and Qualifiers**: Jonatan Männchen raised a need for a format in VERS to express constraints between two version-constraints entries, which is often required in package managers (like Elixir) to specify things like "only stable versions". Philippe Ombredanne suggested that a qualifier for the version range, such as one that dictates whether to include pre-release or post-release versions, might be a way forward, but noted that existing ecosystems like PyPI have unique and "weedy" behaviors.
- **Versioning Issues and Non-Standard Ranges**: Jonatan Männchen noted that a global qualifier might not be sufficient because they have seen real-world cases where non-standard ranges are used, such as starting at a pre-release and allowing only specific types of subsequent releases (e.g., all RCs but not alphas or betas). Jonatan agreed to document all corner cases and weirdness in the issue, potentially extracting version parsing constraints from the Elixir public repo code as a base for discussion.
- **Goal to Standardize Versioning on Public Repos**: Philippe Ombredanne stated that a primary goal should be to at least support everything published on the public repo with all the constraints that are imposed there. He noted that the team has previously chosen not to support arcane and seldom-used corner cases, such as "floating ranges" found in NuGet.
- **Initiating Discussion on PURL Tests**: Michael Herzog introduced the topic of the current test structure, noting that they are working with a v0.1 schema and that a backlog of activities exists. The current test system has two groups: `base` tests for compliance with the general specification and type definition (pass/fail), and `advanced` tests.
- **Review of Current Test Types and Definitions**: The current schema includes three test types: `build` (takes components, creates a canonical PURL), `parse` (takes a PURL string, breaks it into components), and `roundtrip` (takes a string, outputs a PURL string). The definition of `roundtrip` has not been well-defined, and the team is considering whether it is truly necessary alongside a proposed new `validation` type.
- **Proposal for Validation Test Type and Error Handling**: A new test type called `validation` is proposed, with the goal of providing a tool to validate a PURL string, performing any operation on the string and outputting a string. The team discussed whether `roundtrip` and `validation` could be merged into a single type, with the general consensus that renaming `roundtrip` to `validation` might be more descriptive and sufficient.
- **Discussion on Enhanced Error Messaging**: The other key question is error handling, which currently only provides binary pass/fail messages and an `expected failure reason`. The proposal suggested a layered messaging system using severity levels: `info`, `warning`, and `error`, and possibly `success`.
- **Standardizing Error Reporting with Codes and Array**: Jonatan Männchen and Salve J. Nilsen suggested using standardized error codes instead of full messages, which would assist maintainers in debugging across different languages and help avoid encoding issues. The consensus moved toward the concept of having an array of error messages with specific codes, which could indicate which segment of the PURL failed and what the severity level is (e.g., error or warning).
- **Next Steps for Test Schema Update**: Michael Herzog agreed to proceed with the concept of validation as the third test type and to develop a new decision document for the array-based, coded error messages. This change would lead to v0.2 of the schema and would require converting existing test cases and providing concrete guidance on how to construct new tests.
- **Future Discussion on Dependency Schema**: Salve J. Nilsen raised a question about discussing a new schema for dependencies that are feature-based rather than package-specific, suggesting it would be a good topic for this forum.
- **Requirement for Feature and Capability Dependencies**: The participants discussed the need for a feature that allows expressing dependencies on capabilities, services, or abstract resources rather than specific packages, which is seen as critical for meeting new requirements such as those from the EU Cyber Resilience Act. The current dependency graph is incomplete, and a mechanism is required to refer to features, like needing a C library or a mail server, without specifying the exact package name or deployment details. Salve J. Nilsen committed to developing a proposal to address this problem.
- **Scope and Complexity of Non-Package Dependencies**: The discussion broadened to include the potential to express dependencies on resources such as network access, disk space, or a certain amount of input/output operations (IOPS) on a file system, although Philippe Ombredanne suggested this might be too complex for the current context. Michael Herzog clarified that the current topic is broader than library dependencies, encompassing resource or service dependencies, and that a simpler example would be needing a SQL-compliant database, regardless of the backend version. Jonatan Männchen provided an additional example from PHP where interface-only packages fulfill a need, which Michael Herzog related to requiring access to an API.
- **Ecosystem Specificity and Resolution of Abstract Needs**: A key concern raised by Philippe Ombredanne is whether this feature will be too specific to certain ecosystems, like RPM or Debian, where such semantics already exist, or if a generic solution can be developed. Salve J. Nilsen proposed using a set of keywords in a new Package-URL (PURL) scheme (like `capa`(abbreviation for capabilities), `requirement`, or `lib schema URL`) to state the need explicitly but leave the resolution to the consumer of the PURL within their specific system context. They suggested that the consuming system, such as a Linux package manager during a build step, would determine the actual package result based on the required dependency, thereby avoiding the need for the PURL to answer the resolution question.
- **Future Steps for the New Dependency Scheme**: The participants agreed that Salve J. Nilsen will lead the discussion on this topic, and it will be prioritized for the agenda of a coming meeting. Michael Herzog noted that the difference between developing a new type (which can be resolved by a package) and a new scheme (which was the original proposal) is significant, and the proposed scheme is conceptually different from a Package-URL because it focuses on abstract needs rather than packages. The group will reconvene in two weeks, following an upcoming TC54-TG2 meeting.
- The meeting was adjourned.
