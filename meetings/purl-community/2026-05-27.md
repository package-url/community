## Notes

- Meeting minutes are being kept and will be published, and the meeting will be recorded with Google Meet video and Gemini "note-taking".
- Our code of conduct (link in agenda above) applies to this meeting.
- Introductions.
- Summary
  - The team aligned on the ISO "fast track" process and a plan to establish a new official news page for updates. A plan to utilize Listmonk for public outbound communications was initiated.
  - The logical test layer was redefined to include "base" and "advanced" categories while replacing "roundtrip" test types with "validation". A decision was made to prioritize adding failing test cases.
  - Discussions regarding extending PURL types to services highlighted concerns about scope and complexity. The group proposed exploring a separate service-specific identifier to prevent overloading existing software identification standards.
- **Meeting Logistics**: Michael Herzog opened the community call by reminding participants that the meeting is under the terms of the TC54 code of conduct and that the session is being recorded for the purpose of creating minutes.
- **ISO "Fast track" Process**: Michael Herzog provided a process update regarding the ISO "fast track", noting that while the process is not particularly fast, they received the necessary answers to their questions and plan to submit the required statement this week.
- **git PURL Type Status and Outbound Communications**: Jon Moroney requested feedback on the git PURL type, specifically regarding test review and human-oriented documentation. Michael Herzog addressed the lack of a formal process for these updates and outlined a plan to use an open-source tool called Listmonk for public outbound newsletters, while acknowledging the need to manage General Data Protection Regulation (GDPR) compliance for that system.
- **PURL Type Policies and News Page**: Michael Herzog noted that there is no formal policy for deprecating a PURL type, although they aim to point users toward modern versions. To announce significant changes, such as the recent adjustments to the npm PURL type for case sensitivity, the team plans to add a news page to the https://www.packageurl.org website to provide a single, official channel for updates.
- **Slack Channel Management**: Michael Herzog discussed the need to organize and potentially merge the various Slack channels used for PURL, noting that they currently maintain a community channel with approximately 50 to 60 active members and an OWASP CycloneDX channel with over 1,000 members.
- **Test Suite Logical Layer**: Michael Herzog provided an update on the test suite, explaining a redefinition of the "logical layer" which involves implementing distinct "base" and "advanced" test categories. The team plans to replace the "roundtrip" test type with a new "validation" test type, which verifies that inputs and outputs are identical.
- **Advanced Test Message Types**: Michael Herzog introduced two new message types for advanced tests: an "update" message type for correcting non-canonical PURL data using PURL type definitions, and an "information" message type to alert users about syntactically correct but potentially relevant values, such as qualifiers or namespaces.
- **Failing Test Cases**: Michael Herzog and John Horan identified a need for more "failing" test cases to balance the repository, as the current set is dominated by passing tests. Michael Herzog emphasized that they remain agnostic regarding how tools handle these failures, noting that different tools may prioritize different error-handling strategies.
- **ABNF Grammar Implementation**: Michael Herzog discussed the need to implement the Augmented Backus-Naur Form (ABNF) grammar as a pending change to the standard, noting that it defines the physical level of the PURL.
- **PURL Type Interchangeability**: Jonatan Männchen asked about the potential to define type equivalence between different PURL types, such as git, github, and golang. Michael Herzog explained that while the PURL specification itself does not manage cross-referencing, their separate open-source project, PurlDB, serves as a database to cross-reference existing PURLs and their metadata.
- **PurlDB Overview**: Michael Herzog committed to presenting PurlDB to the group to demonstrate its functionality, noting that it is currently grant-funded, primarily by the European Union, and focuses on tracking prioritized, commonly used packages to help developers.
- **agentskills PURL Type Proposal**: Michael Herzog mentioned issue 867 (https://github.com/package-url/purl-spec/issues/867), which proposes creating a new PURL type for "agentskills" Markdown files, and invited the community to provide commentary on the pros and cons of this addition.
- **PURL for Services (sid)**: Van Lindberg initiated a discussion on extending the Software Identification (sid) type for PURL to encompass services, noting that they are building representations of artificial intelligence systems that require identifying constituent parts, including hosted components like models or databases.
- **Services in CycloneDX**: Pavel Shukhman noted that within the CycloneDX specification, services are already treated as components, suggesting that using PURL to identify them might not be the most appropriate approach due to the time-dependent nature of services.
- **Service Identifier Considerations**: Van Lindberg argued that using a DNS-based identifier for services could provide a clear method to backtrack to a responsible party, which would be more definitive for services than it is for standard software.
- **Aggregator Relationships**: Van Lindberg and Jon Moroney discussed the concept of "aggregators" using the example of hosted Redis or Valkey services, concluding that even if two services run the same underlying software version, they are distinct services with different providers and thus require separate identifiers.
- **Alternative Service Identifiers**: Michael Herzog proposed a thought experiment to determine if a separate identifier type, distinct from PURL (perhaps something akin to a "SURL"), would be better suited for services to avoid overloading the sid type with commercial and service-specific requirements.
- **SPDX Service Profile**: Jonatan Männchen noted that the Software Package Data Exchange (SPDX) has a service profile currently in development, which they could not find in the version 3.01 documentation, suggesting it is not yet released.
- **Next Steps for Services Discussion**: Michael Herzog concluded the meeting by encouraging the group to continue thinking about whether a new service-specific identifier is necessary, with plans to revisit the discussion and potentially the proposed "service" type in a future call.
- The meeting was adjourned.
