- Introductions.
- Summary
  - A multi-year proposal to establish a working group for increasing PURL adoption was accepted by OWASP, aiming to make PURL the primary identifier for software in CVE records. The working group's creation is critical because NIST has stopped consistently adding CPE identifiers to new CVE records, making machine-readable identification difficult.
  - The first phase of the working group, estimated to take more than 1 year, includes 4 main tasks, with a focus on developing the new Software Identifier (SID) PURL type for commercial software.
  - The group decided to pursue a single PURL type for both open-source and commercial software, requiring early consensus from commercial developers controlling their product namespaces.
  - Tool development is underway for a new validation tool intended to be a better version of Vulnogram, built in Rust to prevent the publication of bad data.
  - The group debated test suite improvements, concluding that they should proceed with an enhanced error and warning messaging format for advanced tests.
- **Meeting Logistics and Code of Conduct**: The meeting commenced, and Michael Herzog noted that it was being recorded and was subject to the rules of the Ecma TC54 code of conduct. Attendees were informed that if they were uncomfortable with the recording, they should discuss offline communication methods. John Horan is tasked with translating the discussion into meeting minutes, which are published online.
- **PURL Enhancement Working Group Proposal**: Tom Alrich reported that they submitted a multi-year proposal on Monday to OWASP to establish a working group focused on increasing the adoption of Package-URL (PURL). The goal is to advance PURL from being an accepted format in CVE records to becoming the primary, or only, identifier option for software. The proposal was accepted by OWASP within five hours of submission, with Steve Springett listed as co-leader.
- **CPE and Identifier Adoption in CVEs**: A major problem in CVE records is the lack of machine-readable identifiers, as the previously used CPE identifiers are no longer being consistently added by NIST. NIST stopped adding CPE names to every new CVE record in April 2024 and is now focusing only on critical vulnerabilities, which means fewer than 25% of future CVEs will have an identifier unless it is PURL. If there is no machine-readable identifier, the CVE record is difficult to find via search.
- **Phase One Tasks of the Working Group**: Tom Alrich detailed that the first phase of the project, which is estimated to take more than a year, has four tasks. The first task is to develop a new PURL type, initially called "scid" (now "sid"), to identify commercial and non-packaged open-source software, but Tom noted the open-source component may need to be separated. The second task, led by Jon Moroney, involves creating a tool described as a "nicer" version of Vulnogram (https://github.com/vulnogram/vulnogram) for use in the CVE process.
- **Collaborative Development of the New PURL Type**: Developing the new PURL type is characterized as less of a technical problem and more of a "cat herding problem" requiring consensus among commercial software developers. Commercial developers must be involved from the start because they will be responsible for distributing the necessary product information and controlling the namespace for their products. Task one will focus on both the technical definition of the type and the procedural details of distributing the information.
- **Tool Development and Validation**: Jon Moroney noted that the tool they are developing is intended to focus heavily on validation to prevent the publication of bad data. Jonatan Männchen observed that Vulnogram currently has partial PURL support, which is positive, and also prevents the submission of invalid PURLs, which is a good feature. Tom Alrich and Jon Moroney confirmed that validation, built in Rust, will be a key part of the new tool's design.
- **Working Group Funding and Remaining Tasks**: The third and fourth tasks of phase one include developing training materials for CNAs on the existing open-source PURL type and establishing a PURL help desk for CNAs to address questions via email within 24 hours. Tom Alrich confirmed that while the working group is accepted by OWASP, it is not yet funded. Funding applications, including an initial OWASP grant, are necessary to begin the work.
- **Update on the Software Identifier (SID) Initiative**: Michael Herzog provided an update, noting that the new software identifier's code name is now SID (Software IDentifier) instead of SCID. The group decided to continue developing a single PURL type for both open-source and commercial software, despite some internal debate about using two separate types. To improve trackability for major decisions, the group is adopting a very lightweight version of the Architectural Design Record (ADR) approach, with documents stored in `docs/decisions/` (https://github.com/package-url/purl-spec/tree/main/docs/decisions) and associated discussions on GitHub.
- **Refinements to the SID Proposal**: The fundamental proposal for SID has two main changes: the name change to SID and a simplification of the namespace concept. The namespace structure is domain-forward, allowing for additional user-defined segments, and multiple techniques will be available to find more information about a PURL, such as a .well-known locator. A voluntary registry will also be available for open-source projects that do not use domains.
- **Qualifier Requirements and Testing Philosophy**: The current answer regarding qualifiers is that they would not be required for the PURL type itself but could be recommended or validated via advanced tests. The group aims to unbundle the problem by separating the core SID concept from variations in namespace and qualifier validation. Michael Herzog believes that a more realistic timeline for agreeing on the SID basics, getting a definition out, and establishing base tests is one to three months, rather than the six to nine months previously suggested for the entire phase one.
- **Canonical Forms and Comparator Ambiguity in VERS**: Immanuel Kunz raised an issue that VERS' use of non-canonical forms, which are expected to be normalized, creates parsing complexity and increased errors. Immanuel suggested, on behalf of a colleague, not allowing the non-canonical form at all. Philippe Ombredanne strongly agreed, suggesting they should avoid the confusion that arose previously with PURL and focus on declaring canonical all the way.
- **Test Suite Messaging and Structure**: Michael Herzog brought up two related issues concerning the test suite: advanced tests and enhanced messaging. The current test suite provides only pass/fail results, and the proposal is to implement “validation test messages” for advanced tests to provide detailed feedback, such as warning when a name space is in mixed case instead of the required lowercase. The fundamental question for the group is whether they should proceed with an enhanced error/warning messaging format for advanced tests.
- **Upcoming Test Suite Improvements**: The proposed improvements to the test suite would require a major change to the test specification, coordinating with other changes to move the schema to version 2. Jonatan Männchen stated they are in favor of adding the capability for validation tests and offered to provide an example pull request for the hex ecosystem. Michael Herzog will work on recasting the issues and PRs into a more concrete discussion document due to the complexity of the current threads.
- **Review of Assignments and Tasks**: Michael Herzog confirmed that Philippe Ombredanne needs to provide feedback on the two issues that Immanuel raised. Additionally, Tom Alrich committed to circulating information about the OWASP working group. Tom noted that they were not on the mailing list for the project. Michael Herzog confirmed that an official mailing list does not yet exist, but they can still share the information, which Michael Herzog offered to help circulate.
- **Discussion on Communication Channels**: The group acknowledged that there is no official mailing list for the project yet, which remains an outstanding item. Michael Herzog suggested using Slack and existing OWASP channels, such as the PURL channel on the OWASP side, to share information. Tom Alrich confirmed they would send the information to Michael Herzog for distribution.
- The meeting was adjourned.
