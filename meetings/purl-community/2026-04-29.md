## Notes

- Meeting minutes are being kept and will be published, and the meeting will be recorded with Google Meet video and Gemini "note-taking".
- Our code of conduct (link in agenda above) applies to this meeting.
- Introductions.
- Summary
  - Development of the ABNF grammar helps formalize PURL parsing and validates test suites against canonical and non-canonical forms. Participants emphasized using this grammar to identify discrepancies between the written specification and actual implementation.
  - The community evaluated requirements for integrating Conda packages by defining 5 specific artifact identifiers to resolve ambiguity. They agreed that proposals for these specification changes should be submitted via formal issues and pull requests.
  - Members decided to accelerate the VERS standard development by alternating meeting topics between VERS and PURL. The group also agreed to implement semantic versioning for PURL 'type' definitions to manage a backlog of 15 pending changes.
- **Meeting Introduction and Code of Conduct Reminder**: Michael Herzog welcomed attendees to the PURL community meeting and reminded them that the session is being recorded to facilitate meeting note creation, though the video and transcripts will not be published. They emphasized that all participants are subject to the TG2/Package-URL code of conduct and cautioned that for discussions requiring privacy, lead time is needed to stop and restart the recording. They then confirmed that the prior meeting minutes were posted in the channel and that corrections are accepted via pull requests.
- **Welcome New Attendees and Agenda Review**: Michael Herzog welcomed Cheng Lee, noting that they had been present before and are working on related efforts. They announced that an agenda was available, and the first topic for discussion was the Augmented Backus-Naur Form (ABNF) grammar.
- **Update on ABNF Grammar Development**: Jan Kowalleck has been leading the effort to create an ABNF grammar for PURL, which is seen as a crucial tool for encoding, decoding, and validating test cases. ABNF was chosen over EBNF or BNF because PURL is defined as a URL or URI, making ABNF the most appropriate fit. The resulting grammar is a strong candidate for inclusion in a second edition of ECMA-427.
- **Current Status of the ABNF Grammar**: Jan Kowalleck explained that the grammar formalizes how to parse and create a PURL, addressing ambiguities in the current specification. Although the ABNF had to redeclare elements instead of reusing existing RFC declarations, the work started almost a year ago and is currently in a draft state, pending review, but is now more accurate following the release of Package-URL version 1.0.
- **Grammar Use for Test Case Validation**: The primary idea behind having the ABNF grammar is to use tools that can work directly from the grammar to validate existing test suite, including inputs (non-canonical form) and expected outputs (canonical form). While initial testing showed that the current grammar already matches a lot of the existing test suite, it revealed that the test suite does not yet include all complete use cases and edge cases.
- **Call for Review and Use of AI in Grammar Development**: Michael Herzog called for more input and review of the grammar, stressing its importance for test cases and implementation tools. Jan Kowalleck confirmed that the grammar creation process involved utilizing machine learning tools and libraries, supporting Michael Herzog’s observation that this specific area of clear specification is suitable for AI assistance.
- **Discussion on Canonical and Non-Canonical PURLs**: John Horan raised questions regarding the ABNF’s capacity to handle all specification rules and the relationship between the proposed "base" (strict) and "advanced" test cases with "purl" (non-canonical) and "purl-canonical" definitions. Jan Kowalleck confirmed that the ABNF defines both `purl` (allowing backward-compatible things like multiple slashes) and `purl-canonical`.
- **Need for Synchronization Between Spec and ABNF**: Michael Herzog noted that if the ABNF process reveals discrepancies in the written specification, those should be adjusted for a second edition to ensure the two are in sync. They suggested that a tight canonical inner circle and a broader non-canonical outer circle are necessary, especially to provide clear feedback to users when submitted PURLs require correction or normalization.
- **Proposal to Tighten Base Specification Rules**: John Horan highlighted an inconsistency in ECMA-427 where normalization, such as disregarding a trailing slash after `pkg:`, is currently baked into the base definition, which does not strike them as a strict test. Michael Herzog agreed that this needs to be tightened up, as constant data adjustment requires every tool to perform the adjustment consistently. John Horan was tasked with proposing a specification change to address this consistency issue.
- **Proposal for Conda Package Specification Changes**: Jaime Rodríguez-Guerra and Cheng Lee presented a proposal concerning the use of PURLs in the Conda ecosystem, noting that the current PURL specification makes certain assumptions that are not ideal for their environment. Their proposal focuses on adjusting the specification to better describe Conda packages that reside in multiple registries or channels.
- **Addressing Multiple Channel Support in Conda PURLs**: The main challenge is the fact that Conda packages can come from thousands of channels, not just those produced by Anaconda, and there is no single registry of these channels. Cheng Lee confirmed that their draft Anaconda Enhancement Proposal (SEP) will outline how to unambiguously construct the channel name and URL from a given PURL.
- **Required Specificity for Unambiguous Conda PURLs**: To unambiguously identify a single Conda artifact, Jaime Rodríguez-Guerra indicated that five items need to be reflected in the PURL: channel name, subdirectory (platform), package name, version, and build string. The primary issue is that the current PURL specification dictates ambiguous defaults, which they are seeking to rectify.
- **Process for Submitting Conda PURL Changes**: Michael Herzog advised that for formal changes to the PURL specification, the Conda community should first submit an issue with a good rationale and supporting project documentation, followed by a Pull Request. They acknowledged that the Conda representatives are on top of the change process, confirming their role as representatives of the community.
- **Update on Requiring Canonical PURL Inputs**: Immanuel Kunz raised a point regarding a PR they created to require canonical PURL inputs in the specification. Philippe Ombredanne stated they would review the PR promptly.
- **Discussion on Conda Versioning and the VERS Standard**: Cheng Lee confirmed that Conda dependencies are generally specified using a name and a version range, but without a channel, which is left to the client to determine based on priority. The group agreed that Conda's version syntax and comparison methods should be specified in the VERS standard.
- **Planning for Increased Focus on VERS Development**: Michael Herzog noted that the pace for the VERS standard needs to increase to achieve a first edition this year. To better focus the VERS discussion, the participants agreed to try alternating the focus of community meetings between VERS and PURL topics, starting with a focus on VERS for the next meeting.
- **Consideration for Versioning PURL Type Definitions**: Michael Herzog raised the question of whether PURL type definitions should be versioned, noting that a large number of deferred changes are accumulating for existing types (like Go) and substantial changes are being proposed (like Conda). Jan Kowalleck agreed this would be helpful for new type proposers and to track changes over time.
- **Need for Tagged Releases of PURL Type Changes**: Michael Herzog concluded that versioning the PURL type definitions, potentially using a semantic versioning approach, is likely necessary to communicate changes clearly to consumers, especially as there is a backlog of 12 to 15 changes pending. They will tag everything as 1.0 to start and apply increments from there.
- **Open JDK PURL Type Discussion**: Philippe Ombredanne reported on the Eclipse OCX conference where there was interest in a PURL type for Open JDK, given the complexity of multiple Open JDK distributions. Michael Herzog suggested coordinating across the open JDK community to determine if a generic Open JDK type or a more specific one is appropriate.
- **Version Management for Git-Based Academic Software**: Cheng Lee mentioned that some academic software supported by Conda uses a Git commit instead of a formal version number, which presents a challenge for identification. Jon Moroney commented that the development of the Git PURL type is intentionally avoiding versions for this reason and is focusing on the commit and file hash.
- The meeting was adjourned.
