- **Meeting Logistics and Minutes** Michael Herzog welcomed attendees to the PURL community meeting and noted that Philippe Ombredanne was unavailable, so they would be running the meeting. They stated that the meeting is being recorded to generate minutes, which are the official record, and recordings will be kept for about a month. Michael Herzog requested that John Horan post the location of the minutes in the chat, confirming that minutes for the last two meetings are available. They also confirmed that the group will use lightly edited minutes as the record, with confirmation from Steve Springett that Ecma does not require posting recordings to YouTube.
- **Agenda Setting** Michael Herzog proposed a website update and a discussion about tests for the agenda. Matt Rutkowski added a topic to share their work on the machine learning bill of materials (ML-BOM) guide related to PURL. Jon proposed discussing the Version Control System (VCS) PURL type, specifically whether it should be a single type, multiple types, or how to approach it. Steve Springett added the scid type to the backlog for a future meeting with Philippe, to talk about unifying the different proposals to avoid fragmentation.
- **Website Update and Milestones** Michael Herzog announced that the live website, http://www.packageurl.org, is up and running on DreamHost as release one, although it has not yet been formally tagged. They also mentioned that `www.package-url.org` will be set up to redirect to the new site, and GitHub will continue to be used for staging. John Horan noted that they are using GitHub actions and a YAML file for the process. Michael Herzog confirmed that the website does not perform any data collection or use trackers.
- **Community Contributions and Tooling** Michael Herzog reported creating early issue templates for people wanting to add their tools to the tool grid or specifications to the spec grid/adopters' side. They also mentioned an upcoming issue template for adding a PURL type, which involves considerations like obtaining logos or icons. They noted that the vscode-extension type was approved, and the Homebrew type PR seems to be in good shape, though testing for new additions is a concern. Michael Herzog is looking to engage with students for website contributions and mentioned that a contributor named Sebastien Cellis created a new PURL viewer tool. Steve Springett reviewed the homepage using Lighthouse and reported good numbers, with the main opportunity being security headers.
- **Specification Document Reorganization (vers-spec)** Michael Herzog shared that they and John Horan completed a reorganization of the VERSION-RANGE-SPEC.md file in the docs directory of the vers-spec repo, mirroring the structure used for PURL spec. The new structure includes a document attempting to distill the essence of the specification. The goal is to prepare a "skinny but impactful" submission to Ecma in March, though Michael Herzog noted that Philippe has not yet reviewed the reorganized document.
- **Machine Learning Bill of Materials (ML-BOM) Guide** Matt Rutkowski presented their work on the ML-BOM guide, which was restarted with Nvidia. The guide incorporates requirements primarily from the EU AI Act and addresses what should be included in a model card within CycloneDX. Matt Rutkowski stated that the guide is about 96% complete and will come off draft status soon.
- **Package-URL Usage in ML-BOM Guide** Matt Rutkowski emphasized the proper referencing and promotion of Package-URL (PURL) within the ML-BOM guide, particularly the use of the Hugging Face PURL type to define machine learning model componentry. They also introduced the recommendation to create a BOM reference (`bomb-ref`) based on the PURL, specifically using the first seven characters of the commit hash to avoid bloating the bill of materials. Steve Springett confirmed that they were using the GitHub PURL type for the ONNX model as an existing type.
- **ML-BOM Guide Promotion and Taxonomy** Matt Rutkowski outlined plans to promote the ML-BOM guide work through presentations to groups like the OpenSSF and Alpha Omega project, with Steve Springett potentially serving as a conduit to the OWASP Gen AI project. Matt Rutkowski also created an ML property taxonomy to describe models, tokenizers, and prompts, which can be extended. They showed an example of how a custom tokenizer can be called out using a PURL within the model template.
- **Discussion of VCS PURL Types** Jon initiated the discussion on the VCS PURL type, outlining three approaches: a single generic type with subtypes, first-class types for every registry, or types for version control systems (like Git, Mercurial). Jon favors a "git type" to handle cases where GitHub is not the host, allowing tools to assert that it is a Git repository. Steve Springett suggested adopting the Maven SCM (Source Code Management) URL concept as a primitive type, noting that millions of Java projects already use this format, which could ease conversion to a Package-URL.
- **Proposal for SCM/VCS PURL Types** Jon and Michael Herzog discussed a proposal for a hybrid approach where specific, popular types (like GitHub) are maintained, alongside a more generic SCM or VCS type to cover the remaining cases. The generic type would serve as a "fallback" or "stop gap". Jon suggested that new, popular registries could propose a proper PURL type for approval. Michael Herzog agreed that this practical approach allows tool developers to implement something recognizable immediately and also introduces the concept of recognizing legacy types for compatibility, without the need to cancel old ones.
- **CVSS Records and PURL Adoption** Jon shared that there are now 105 occurrences of PURLs in the CVE record corpus, with one instance coming from MITRE, the creator of CPE. Steve Springett viewed this as a huge step, especially since the record found by Jon did not contain a CPE, highlighting significant progress. Michael Herzog noted that this adoption also serves as a critical example of why the version field cannot be required in PURL, as it is often not available in vulnerability use cases.
- **Testing System Update and Test Types** Michael Herzog provided an update on the test system, stating they are rewriting the test system and will publish a file outlining the current state. They plan to discuss this work at the next TG2 meeting, as the testing process requires the most review for new PURL types. Michael Herzog questioned the distinction between the "round-trip" and "validation" test types, suggesting that less is more and proposing that validation could simply involve enriched messages rather than being a separate test type.
- **VERS Specification Review** Steve Springett requested an update on progress for the VERS specification, and Michael Herzog encouraged Steve Springett to review the specification document located at https://github.com/package-url/vers-spec/blob/main/docs/standard/specification.md in the vers-spec repo before Philippe's review. Michael Herzog noted difficulties decoding Philippe's information regarding the pipe '|' separators (AND/OR concepts). They also mentioned that they are deliberately holding off on a community-wide review until Philippe has had a chance to look at it .
- The meeting was adjourned.
